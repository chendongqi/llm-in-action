# 微服务架构设计指南

本文介绍如何设计和实现一套生产级的微服务架构，涵盖服务拆分、通信协议、数据一致性、可观测性等核心主题。

---

## 1. 服务拆分策略

### 1.1 按业务边界拆分（DDD）

领域驱动设计（Domain-Driven Design, DDD）是微服务拆分的黄金法则。核心思想是：每个服务对应一个业务子域（Subdomain），服务之间的边界就是业务边界。

例如，一个电商平台可以拆分为：
- **用户服务**：负责用户注册、登录、个人信息管理
- **商品服务**：负责商品目录、SKU管理、库存查询
- **订单服务**：负责订单创建、状态流转、取消逻辑
- **支付服务**：负责支付渠道对接、对账、退款
- **物流服务**：负责运单生成、物流跟踪、配送调度

拆分的原则是高内聚、低耦合。如果一个功能需要同时修改三个服务，说明拆分粒度可能过细；如果一个服务内部有多个互不相关的模块，说明粒度可能过粗。

### 1.2 按技术特性拆分

除了业务边界，也可以按技术特性拆分：
- **读服务 vs 写服务**：读多写少的场景，读写分离可以独立扩缩容
- **实时服务 vs 批处理服务**：实时交易和离线报表对资源的需求完全不同
- **核心服务 vs 边缘服务**：核心服务需要更高的可用性保障

### 1.3 拆分的常见反模式

**分布式单体**：表面上是多个服务，实际上所有服务共享同一个数据库，服务之间通过数据库隐式耦合。任何表结构变更都需要协调所有服务发布。

**过度拆分**：一个只有5个工程师的团队拆出了20个服务。结果是每个工程师要维护4个服务，上下文切换成本极高，且无法保证每个服务的质量。

**纳米服务**：比微服务更细粒度的拆分，例如把"验证手机号"和"验证邮箱"拆成两个独立服务。网络调用的开销远大于业务逻辑本身。

## 2. 服务间通信

### 2.1 同步通信：REST vs gRPC

微服务之间最常见的同步通信方式是 HTTP REST 和 gRPC。

**REST** 基于 HTTP/1.1 或 HTTP/2，使用 JSON 作为序列化格式。优点是简单、调试方便、人类可读。缺点是 JSON 序列化开销大，HTTP/1.1 有队头阻塞问题。

**gRPC** 基于 HTTP/2 和 Protocol Buffers。优点是：
- 二进制序列化，性能比 JSON 高 5~10 倍
- 支持流式通信（客户端流、服务端流、双向流）
- 强类型接口定义，自动生成客户端代码

缺点是调试相对困难，需要工具如 grpcurl 或 BloomRPC。

选型建议：内部服务间通信用 gRPC，对外暴露的 API 用 REST（兼容性和可读性更好）。

### 2.2 异步通信：消息队列

异步通信通过消息队列解耦服务，常见的有 Kafka、RabbitMQ、RocketMQ。

**Kafka** 适合高吞吐量的日志收集和事件流处理。Topic 分区机制支持水平扩展，但延迟相对较高（毫秒级）。

**RabbitMQ** 适合需要复杂路由规则的场景。支持 Exchange 绑定、死信队列、延迟队列等高级特性，但单节点吞吐量不如 Kafka。

**RocketMQ** 是阿里开源的消息队列，兼顾了 Kafka 的高吞吐和 RabbitMQ 的丰富特性，在国内使用广泛。

消息队列的核心优势是削峰填谷。例如，电商大促时订单服务可能瞬间收到 10 万笔订单，如果直接调用支付服务，支付服务会被压垮。通过消息队列，订单服务把订单事件写入队列，支付服务按自己的处理能力消费，系统整体更稳定。

### 2.3 通信模式选择决策树

```
需要立即得到响应？
    ├─ 是 → 同步通信（REST/gRPC）
    │         需要高性能？ → gRPC
    │         需要兼容性好？ → REST
    └─ 否 → 异步通信（消息队列）
              需要高吞吐？ → Kafka
              需要复杂路由？ → RabbitMQ
              国内生态？ → RocketMQ
```

## 3. 数据一致性

### 3.1 分布式事务的挑战

单体应用里，一个数据库事务可以保证 ACID。微服务架构下，每个服务有自己的数据库，跨服务的事务就变成了分布式事务。

分布式事务的经典问题是"两阶段提交"（2PC）的缺陷：协调者单点故障、同步阻塞性能差、脑裂问题。因此，微服务架构中通常避免使用 2PC。

### 3.2 Saga 模式

Saga 模式把一个大事务拆成一系列本地事务，每个本地事务完成后发送事件触发下一个。如果某个步骤失败，执行补偿操作（回滚）。

例如，下单流程的 Saga：
1. 创建订单（本地事务）→ 发送"订单已创建"事件
2. 扣减库存（本地事务）→ 发送"库存已扣减"事件
3. 扣减余额（本地事务）→ 发送"余额已扣减"事件
4. 创建物流单（本地事务）→ 完成

如果步骤 3 失败，执行补偿：
- 补偿步骤 2：恢复库存
- 补偿步骤 1：取消订单

Saga 有两种实现方式：
- **编排式（Choreography）**：每个服务完成自己的事务后，发送事件，由下游服务监听并执行。优点是去中心化，缺点是事件链复杂时难以追踪。
- **编排式（Orchestration）**：由一个协调器服务统一调度各个步骤。优点是逻辑集中、易于监控，缺点是协调器成为单点。

### 3.3 最终一致性 vs 强一致性

微服务架构默认接受**最终一致性**。这意味着，在某一时刻，不同服务看到的数据可能不一致，但经过一段时间后最终会一致。

例如，用户下单后，订单服务立即返回"下单成功"，但库存服务和支付服务可能有几毫秒的延迟才更新状态。对于绝大多数业务场景，这种延迟是可以接受的。

只有在极少数场景（如金融转账）才需要强一致性，这时可以考虑：
- 把强一致性相关的逻辑放在同一个服务内
- 使用分布式锁（Redis RedLock、ZooKeeper）
- 使用支持分布式事务的数据库（如 TiDB、OceanBase）

## 4. 可观测性

### 4.1 三大支柱：日志、指标、追踪

可观测性（Observability）是微服务运维的核心能力。Google SRE 理论将其归纳为三大支柱：

**日志（Logging）**：记录系统运行时的离散事件。例如，"2024-05-01 10:23:45 用户 12345 下单成功，订单号 67890"。日志适合排查特定问题，但不适合做聚合分析。

**指标（Metrics）**：记录系统的量化状态。例如，CPU 使用率 75%、请求 QPS 1200、错误率 0.5%。指标适合监控和告警，是容量规划和自动扩缩容的基础。

**追踪（Tracing）**：记录请求在多个服务间的完整调用链。例如，一个用户请求经过网关 → 订单服务 → 库存服务 → 数据库，追踪可以展示每个环节的耗时和状态。分布式追踪的代表工具是 Jaeger 和 Zipkin。

### 4.2 统一日志收集

微服务环境下，日志分散在几十个甚至上百个容器中，手工查看不现实。需要统一的日志收集方案：

**ELK 栈**（Elasticsearch + Logstash + Kibana）：经典的日志收集方案。Filebeat 采集日志 → Logstash 解析过滤 → Elasticsearch 存储 → Kibana 可视化。

**Loki**（Grafana 出品）：轻量级的日志收集方案。与 ELK 相比，Loki 只索引日志的标签（metadata），不索引日志内容，存储成本更低，查询性能在大部分场景下足够。

日志收集的关键实践：
- 所有服务使用统一的日志格式（JSON 推荐）
- 每个日志条目包含 trace_id，方便关联分布式追踪
- 日志分级：ERROR 立即告警，WARN 定期审查，INFO 用于排查，DEBUG 仅开发环境开启

### 4.3 告警策略

告警不是越多越好。过多的告警会导致"告警疲劳"，运维人员会习惯性地忽略告警。

好的告警策略：
- **基于症状而非原因**："用户下单成功率低于 99%" 比 "CPU 使用率超过 80%" 更有业务价值
- **分层告警**：P0（立即处理，如服务完全不可用）、P1（30分钟内处理，如错误率升高）、P2（工作时间内处理，如磁盘空间不足）
- **告警收敛**：同一个根因引发的多个告警，合并为一条通知

## 5. 安全设计

### 5.1 服务间认证

微服务之间需要互相认证，防止未授权的服务访问敏感接口。常见的方案：

**mTLS（双向 TLS）**：服务之间通过证书互相验证身份。Istio、Linkerd 等服务网格自动为服务注入 mTLS，无需修改业务代码。

**JWT + API Gateway**：外部请求通过 API Gateway 认证，Gateway 把 JWT Token 转发给下游服务。内部服务之间可以选择信任 Gateway（不二次验证）或独立验证 JWT。

### 5.2 敏感数据脱敏

日志和追踪中不能包含敏感数据（密码、身份证号、银行卡号）。需要在日志输出层做脱敏处理：

```python
# 脱敏示例
import re

def mask_phone(phone: str) -> str:
    return re.sub(r'(\d{3})\d{4}(\d{4})', r'\1****\2', phone)

def mask_idcard(idcard: str) -> str:
    return idcard[:6] + '********' + idcard[-4:]
```

### 5.3 最小权限原则

每个服务只应该拥有完成其职责所需的最小权限：
- 订单服务只能访问订单表，不能访问用户表
- 支付服务只能访问支付渠道 API，不能访问物流系统
- 数据库账号按服务隔离，不允许共享 root 账号

## 6. 部署与运维

### 6.1 容器化与编排

Docker 容器是微服务的标准交付单元。Kubernetes 是容器编排的事实标准，提供：
- **服务发现与负载均衡**：Service 和 Ingress 自动分发流量
- **自动扩缩容**：HPA（水平 Pod 自动扩缩容）根据 CPU/内存/QPS 自动调整副本数
- **滚动更新**：Deployment 的 RollingUpdate 策略实现零停机发布
- **健康检查**：LivenessProbe 和 ReadinessProbe 自动剔除异常实例

### 6.2 配置管理

微服务的配置分散且频繁变更，需要集中管理：

**环境变量**：最简单的配置方式，适合不敏感的小规模配置。缺点是配置变更需要重启服务。

**配置中心**：Spring Cloud Config、Nacos、Apollo 等。支持动态刷新（不重启服务）、配置版本管理、灰度发布。是国内中大型微服务架构的标配。

**GitOps**：把配置放在 Git 仓库中，通过 CI/CD 流水线自动同步到集群。优点是版本可控、变更可审计、回滚简单。

### 6.3 蓝绿发布与金丝雀发布

微服务架构下，频繁发布是常态。需要安全的发布策略：

**蓝绿发布**：同时部署两套环境（蓝环境和绿环境），切换流量时瞬间完成。优点是回滚快，缺点是资源消耗翻倍。

**金丝雀发布（Canary）**：先让 5% 的流量访问新版本，观察指标无异常后逐步扩大到 100%。优点是风险可控，缺点是需要精细的流量控制能力和监控能力。Istio 的 VirtualService 和 Flagger 是实现金丝雀发布的常用工具。

## 7. 总结

微服务架构不是银弹。在决定拆分单体应用之前，需要评估团队的技术能力、运维能力和业务复杂度。

适合微服务的团队特征：
- 团队规模 20 人以上，可以分成多个小团队（2 Pizza Team）
- 业务复杂度较高，不同模块的发布节奏差异大
- 有专职的 DevOps/SRE 团队负责基础设施

不适合微服务的情况：
- 团队规模小于 10 人，维护多个服务的成本过高
- 业务逻辑简单，单体应用足以满足需求
- 没有容器化基础设施和自动化运维能力

记住：**微服务解决的是组织协作问题，不是技术性能问题。** 如果团队协作顺畅，单体架构可以支撑很大的业务规模（如 Shopify 的单体 Ruby on Rails 应用）。
